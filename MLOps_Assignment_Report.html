1. Project Overview

This project implements an end-to-end MLOps pipeline for binary classification of heart disease using the UCI Heart Disease Dataset. The pipeline covers every stage of the ML lifecycle: data acquisition, exploratory data analysis, preprocessing, model training with hyperparameter tuning, experiment tracking (MLflow), model packaging, REST API serving (FastAPI), containerisation (Docker), Kubernetes deployment, CI/CD automation (GitHub Actions), and production monitoring (Prometheus + Grafana).

Objectives

  • Build a reproducible, production-grade ML pipeline following MLOps best practices.
  • Train and compare four classifiers; select the best by ROC-AUC.
  • Expose the model via a versioned REST API with health checks and Prometheus metrics.
  • Automate the entire workflow with GitHub Actions CI/CD.
  • Deploy to Kubernetes with liveness/readiness probes and resource limits.

2. Setup & Installation Instructions

Prerequisites

  • Python 3.11+
  • Docker & Docker Compose
  • kubectl + Minikube (optional, for Kubernetes deployment)
  • Git

Step-by-step Installation

1. Clone the repository

git clone https://github.com/Sunil-Kumar-2024ac05659/MLOps-Assign-1.git
cd MLOps-Assign-1

2. Create a virtual environment and install dependencies

python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt

3. Download the dataset

python data/download_data.py

6. Run the REST API

uvicorn api.main:app --reload --port 8000
# Swagger docs: http://localhost:8000/docs

7. Run tests

pytest tests/ -v --cov=src --cov=api

Key EDA Findings

  • thalach (max heart rate) shows the strongest negative correlation with disease — patients with disease tend to have lower max heart rates.
  • oldpeak (ST depression) shows positive correlation with disease presence.
  • cp (chest pain type) is highly discriminative — type 0 (typical angina) strongly associated with disease.
  • exang (exercise-induced angina) is a strong categorical predictor.
  • chol shows weak correlation with the target, suggesting limited predictive power as a standalone feature.
  • Class balance is near 54/46 — no resampling required; stratified splits used.
  • Age distribution is right-skewed; older patients appear at higher risk.

4. Feature Engineering & Modelling Choices

Preprocessing Pipeline

A scikit-learn ColumnTransformer is used to handle numerical and categorical features independently, inside a full Pipeline to prevent data leakage.

StepNumerical FeaturesCategorical Features
ImputationMean imputation (SimpleImputer(strategy="mean"))Most-frequent imputation
Scaling / EncodingStandardScaler (zero mean, unit variance)OneHotEncoder (handle_unknown="ignore")

Train / Test Split

  • Split ratio: 75% train / 25% test
  • Stratified on target class to preserve class balance in both sets
  • Random state: 7
  • Training samples: 227  |  Test samples: 76

Cross-Validation Strategy

10-fold Stratified K-Fold cross-validation (random_state=7) with GridSearchCV optimising for roc_auc. After tuning, a second cross_validate call on the best estimator records mean ± std for accuracy, F1, and ROC-AUC on the training set.

5. Experiment Tracking Summary (MLflow)

MLflow is used for full experiment tracking. Every GridSearchCV run is wrapped in mlflow.start_run(). The following are recorded automatically:

  • Parameters: best hyperparameters from grid search (e.g. C, solver, n_estimators, max_depth, learning_rate)
  • Metrics: CV mean/std for accuracy, precision, recall, F1, ROC-AUC; test-set values for all five metrics
  • Artifacts: confusion matrix PNG, ROC curve PNG, feature importance PNG, serialised sklearn pipeline
  • Tags: model_name tag on each run for easy filtering

Experiment Configuration

SettingValue
Experiment nameheart-disease-classifier
Tracking URILocal filesystem (mlruns/)
CV folds10-fold Stratified K-Fold
Scoring metricroc_auc
Total runs4 (one per model)

Launching MLflow UI

mlflow ui --port 5000
# Navigate to: http://localhost:5000

The MLflow UI provides a side-by-side comparison of all four runs, sortable by any metric. Artifact viewers allow inspection of plots and the serialised model directly in the browser.

6. Model Packaging & Reproducibility

Model Serialisation

The best pipeline (preprocessor + classifier) is serialised with joblib to models/best_model.joblib. The full sklearn Pipeline object is saved, which means the same preprocessing steps used at training time are automatically applied at inference — preventing any train/serve skew.

Model Metadata

A models/model_metadata.json file is written alongside the model:

{
  "model_name": "LogisticRegression",
  "metrics": {
    "accuracy": 0.9211,
    "precision": 0.9394,
    "recall": 0.8857,
    "f1": 0.9118,
    "roc_auc": 0.9554
  },
  "num_features": ["age", "trestbps", "chol", "thalach", "oldpeak"],
  "cat_features": ["sex", "cp", "fbs", "restecg", "exang", "slope", "ca", "thal"]
}

Reproducibility Measures

  • All random states fixed: RANDOM_STATE = 7
  • Pinned dependency versions in requirements.txt
  • Deterministic data split via train_test_split(..., random_state=7, stratify=y)
  • Pipeline includes preprocessor + model in a single joblib artifact
  • MLflow logs every run's parameters, metrics, and artifacts for full audit trail
  • Model also registered via mlflow.sklearn.log_model() for MLflow Model Registry compatibility

7. API Design

The model is served via a FastAPI application (api/main.py). The model pipeline is loaded once at startup using FastAPI's lifespan context manager.

Endpoints

MethodEndpointDescription
GET/healthLiveness check; returns model load status
GET/metricsPrometheus metrics (text format)
POST/predictSingle-record heart disease prediction
POST/predict/batchBatch prediction (list of patients)
GET/docsAuto-generated Swagger UI

Request Schema (POST /predict)

curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "age": 63, "sex": 1, "cp": 3, "trestbps": 145, "chol": 233,
    "fbs": 1, "restecg": 0, "thalach": 150, "exang": 0,
    "oldpeak": 2.3, "slope": 0, "ca": 0, "thal": 1
  }'

Response Schema

{
  "prediction": 1,
  "label": "Heart Disease",
  "confidence": 0.8712,
  "probabilities": {
    "no_disease": 0.1288,
    "disease": 0.8712
  }
}

API Design Decisions

  • Pydantic v2 validation ensures all input fields are typed and bounded before inference.
  • 503 response when model is not loaded (e.g. missing best_model.joblib).
  • Structured logging via Python's logging module with timestamp, level, and module prefix.
  • Optional features (slope, ca, thal) allow partial inputs; pipeline imputer handles missing values at serve time.

9. CI/CD Pipeline

The CI/CD pipeline is defined in .github/workflows/ci-cd.yml and triggers on every push or pull request to main / develop.

Pipeline Stages

StageToolWhat it does
1. Lint flake8 Static analysis on src/, api/, tests/. Max line length 120. Fails fast before tests run.
2. Test pytest + pytest-cov Unit and integration tests with coverage report (XML + terminal). Uploads coverage.xml as artifact. Needs lint to pass.
3. Train python src/train.py Downloads dataset, trains all 4 models, logs to MLflow. Uploads best_model.joblib, model_metadata.json, and mlruns/ as artifacts. Needs tests to pass.
4. Docker Build & Test Docker CLI + curl Builds Docker image, runs container, waits for /health readiness, tests /predict endpoint. Collects logs on failure. Needs training to pass.

Pipeline YAML (key excerpt)

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint   → test → train → docker-build   # sequential dependency chain

  # Stage 3: Train
  - name: Download dataset
    run: python data/download_data.py
  - name: Train models
    run: python src/train.py
  - name: Upload trained model artifact
    uses: actions/upload-artifact@v4
    with:
      name: trained-model
      path: |
        models/best_model.joblib
        models/model_metadata.json

10. Containerisation & Deployment

Docker

Base image: python:3.11-slim. Layers are ordered for cache efficiency (dependencies installed before source copy).

# Build
docker build -t heart-disease-api:latest .

# Run standalone
docker run -p 8000:8000 heart-disease-api:latest

# Full stack (API + Prometheus + Grafana)
docker-compose up --build

Kubernetes Deployment

Deployment manifest (k8s/deployment.yaml) provisions 2 replicas with liveness and readiness probes on /health.

# Start Minikube
minikube start

# Build image inside Minikube
eval $(minikube docker-env)
docker build -t heart-disease-api:latest .

# Deploy
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

# Get URL
minikube service heart-disease-api-service --url

11. Monitoring & Logging

Prometheus Metrics

Three custom metrics are exposed at GET /metrics (scraped by Prometheus every 15s):

Metric NameTypeLabelsDescription
api_requests_totalCounterendpoint, method, statusTotal API requests by endpoint, method, and HTTP status
api_request_latency_secondsHistogramendpointRequest latency in seconds per endpoint
predictions_totalCounterlabelTotal predictions made, broken down by "Heart Disease" / "No Heart Disease"

Prometheus Scrape Config

global:
  scrape_interval: 15s

scrape_configs:
  - job_name: "heart-disease-api"
    static_configs:
      - targets: ["api:8000"]
    metrics_path: /metrics

Grafana

Grafana is included in the Docker Compose stack, accessible at http://localhost:3000. Add Prometheus as a data source (http://prometheus:9090) and build dashboards from the three metric series above. Useful panels:

  • Request rate per endpoint (rate of api_requests_total)
  • P95/P99 latency histogram (from api_request_latency_seconds)
  • Prediction class distribution over time (predictions_total)
  • Error rate (status=5xx / total requests)

Application Logging

Python's standard logging module is configured at INFO level with the format timestamp | LEVEL | module | message. Key events logged:

  • Model load at startup (path + success/failure)
  • Each prediction (label + confidence)
  • Batch prediction summary (record count)
  • Prediction errors (exception message)
  • API shutdown
